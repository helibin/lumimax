# 第三方食品营养识别调研（最终研发落地版）

> 本文是阶段一 MVP 的研发执行版本，融合 `第三方食品库接入调研.md` 与 `food-provider-strategy.md` 的结论，并明确**国内 / 国外两条独立调研链路**。  
> 如与其他文档冲突，以 `分阶段路线图.md` 与 `设备接入协议规范v1.3.md` 最新版本为准。

---

## 1. 目标与范围

阶段一 MVP 目标：在设备端跑通以下闭环并稳定上线。

```text
拍照识别 -> 候选展示 -> 设备确认 -> 营养估算 -> 多次记录 -> 本餐汇总
```

范围约束：

- **国内、国外并行上线**，每个区域**独立部署**、独立服务、独立 provider 配置；**不做跨境调用**。
- 多目标识别必须支持 `name + confidence + count`，重量统一以设备秤为准。
- 单次 `food.analysis.request` 直接返回**预估营养**用于设备 UI；本餐结束再返回最终汇总。
- 不做家庭账号体系、不做实时差值动画（S7）、不做实时称重事件流。

---

## 2. 架构原则

### 2.1 三段职责

| 段位 | 职责 | 必选输出 |
| --- | --- | --- |
| 视觉识别层 | 给候选（是什么、几个、置信度） | `name + confidence + count` |
| 营养查询层 | 给标准营养并按重量换算 | `calories + nutrition + source` |
| 设备确认层 | 确认/修正后沉淀 | `user_common` 更新 |

### 2.2 四条硬规则

- AI 只提供候选，不直接作为最终事实。
- 重量只信任设备称重，不信视觉估重。
- 营养来源必须可追溯（`source` + `confidence`）。
- 仅在设备/用户确认后写 `user_common`，否则只走临时缓存。

### 2.3 区域隔离原则

- 国内/国外是**两套独立配置**，包括 vision provider、nutrition provider、缓存 key、外部网络出口。
- 同一份代码通过环境变量（`REGION=cn` / `REGION=us`）切换 provider 池。
- 不做跨区数据合并；`user_common` 在各自区域内沉淀。

## 3. 调研方（Provider 池总览）

> 这一节列举我们已经评估过的、可纳入 MVP 或下一步的所有 Provider，并标注定位与适用区域。具体使用顺序见 §4 调研链路。

### 3.1 视觉识别 Provider

| Provider | 类型 | 区域适配 | 阶段一定位 | 备注 |
| --- | --- | --- | --- | --- |
| OpenAI Vision (GPT-4o / GPT-4o-mini) | LLM 多模态 | us 首选 | MVP 首选（us） | prompt 强制输出 `count + confidence` |
| Gemini Vision (1.5 Pro / Flash) | LLM 多模态 | us 备选 | MVP 备选（us） | 失败时自动 fallback |
| Qwen-VL / Qwen2-VL | LLM 多模态 | cn 首选 | MVP 首选（cn） | 国内出口、合规友好 |
| DeepSeek Vision | LLM 多模态 | cn 备选 | MVP 备选（cn） | 价格优势 |
| Passio Nutrition-AI | 专用视觉 SDK | 全球 | 长期评估 | 含条码、文本、营养结构 |
| FatSecret Image Recognition | 专用视觉 API | 全球 | 长期评估 | 直接映射到本地 verified 库 |
| LogMeal | 专用视觉 API | 全球 | 长期备选 | 仅识别能力 |

### 3.2 营养查询 Provider

| Provider | 数据特点 | 区域适配 | 阶段一定位 | 接入方式 |
| --- | --- | --- | --- | --- |
| `user_common` | 用户/设备沉淀的历史食材 | cn / us | **必选**（最高优先） | 本地表 |
| `boohee`（薄荷健康） | 国内常见食材/餐饮 | cn | **必选**（cn 主补充） | HTTP API |
| `cn_internal` | 中国食物成分表 + 自建条目 | cn | 必选（cn 基础库） | 本地数据 / 离线 |
| `usda_fdc` | USDA FoodData Central，基础食材权威 | us / cn 兜底 | **必选**（us 基准） | REST API |
| `edamam` | 文本/食谱解析，28+ 营养项 | us | 建议接入 | REST API |
| `open_food_facts` | 全球包装食品/条码 | us / cn 可选 | 可配置 | REST API |
| `nutritionix` | 北美品牌/餐厅食品 | us | 弱依赖 | REST API |
| `fatsecret` | 全球商业主库 | us 长期 | 长期评估 | REST API（需商务） |
| `llm_estimated` | 模型估算兜底（OpenAI/Gemini/Qwen/DeepSeek） | cn / us | **必选**（最低优先） | 通过 env 切换 |

> 命名规范：所有 `source` 枚举值与协议字段保持一致，见 `设备接入协议规范v1.3.md`。

## 4. 调研链路（按区域拆分）

> 调研链路 = 一次单品识别请求的完整 provider 调用顺序。任一上层命中即返回，**不再继续向下查**；命中时携带的 `source` 即写入设备协议。

### 4.1 国内（CN）调研链路

**部署形态**：国内云 + 国内出口；vision 使用国内模型，避免跨境拉取图像。

**链路顺序**：

```text
[1] 视觉识别
    Qwen-VL（首选） → DeepSeek Vision（兜底）
    输出：items[] = { name, canonicalName, count, confidence }

[2] 营养查询（按 canonicalName 逐项回查）
    user_common         （设备维度的历史命中）
        ↓ miss
    cn_internal         （中国食物成分表 + 自建词条）
        ↓ miss
    boohee              （薄荷健康，国内主补充库）
        ↓ miss
    open_food_facts     （仅条码/包装场景，默认关）
        ↓ miss
    llm_estimated       （Qwen / DeepSeek，低置信兜底）

[3] 设备确认
    food.analysis.confirm.request
        → 写 meal_item
        → 命中后回填 user_common
```

**国内场景设计要点**：

- **只识别原材料**：MVP 不识别成品菜（番茄炒蛋 → 视为「番茄 + 鸡蛋」分别上称），降低误差。
- 视觉 prompt 强制返回 `count`（如「番茄 2 个，鸡蛋 3 个」），重量由秤体上报。
- `boohee` / `cn_internal` 命中时优先返回，避免触发模型估算。
- 出口仅指向国内服务，不调用 `usda_fdc` / `edamam` / `nutritionix`（除非显式开启）。

### 4.2 国外（US）调研链路

**部署形态**：海外云 + 海外出口；vision 使用国外模型，可识别成品菜。

**链路顺序**：

```text
[1] 视觉识别
    OpenAI Vision（首选） → Gemini Vision（兜底）
    输出：items[] = { name, canonicalName, count, confidence }

[2] 营养查询（按 canonicalName 逐项回查）
    user_common         （设备维度的历史命中）
        ↓ miss
    usda_fdc            （USDA FoodData Central，基础食材权威）
        ↓ miss
    edamam              （文本/食谱解析，覆盖菜品）
        ↓ miss
    open_food_facts     （条码/包装食品场景）
        ↓ miss
    nutritionix         （北美品牌/餐厅食品补充）
        ↓ miss
    llm_estimated       （OpenAI / Gemini，低置信兜底）

[3] 设备确认
    food.analysis.confirm.request
        → 写 meal_item
        → 命中后回填 user_common
```

**国外场景设计要点**：

- 允许识别成品菜（salad、burger 等），由 `edamam` 文本解析补营养。
- 包装食品场景将来可前置 `open_food_facts`（条码扫描）。
- `nutritionix` 仅在前 4 个 provider 全 miss、且为品牌/餐厅类时启用。
- `fatsecret` 阶段一**暂不接入**，作为长期商业主库候选。

### 4.3 两条链路对照表

| 步骤 | 国内（CN） | 国外（US） |
| --- | --- | --- |
| 视觉首选 | Qwen-VL | OpenAI Vision |
| 视觉兜底 | DeepSeek Vision | Gemini Vision |
| 识别粒度 | 仅原材料 | 原材料 + 成品菜 |
| 营养顺序 | user_common → cn_internal → boohee → off → llm | user_common → usda_fdc → edamam → off → nutritionix → llm |
| 兜底模型 | Qwen / DeepSeek | OpenAI / Gemini |
| 出口 | 国内出口 | 海外出口 |
| 跨区调用 | 禁止 | 禁止 |

## 5. 接口行为要求

### 5.1 单次识别返回（`food.analysis.result`）

每个候选项必须包含：

```text
foodId / name / canonicalName / count / confidence / source
calories / nutrition
```

- 字段容器统一使用 `items[]`（不再使用 `recognitions[]`）。
- `source` 枚举严格对应 §3.2 中表中 Provider 列：`user_common` / `boohee` / `cn_internal` / `usda_fdc` / `edamam` / `open_food_facts` / `nutritionix` / `llm_estimated`。
- `confidence` 区间统一为 `[0, 1]`。`llm_estimated` 必须 `< 0.5`。

### 5.2 设备确认（`food.analysis.confirm.request/result`）

设备在收到 `items[]` 后必须上报确认事件：

- 直接确认某一项
- 或修正：name、count、weightGram 任一字段
- 仅确认后服务端写入 `meal_item` 与 `user_common`

### 5.3 本餐汇总（`nutrition.analysis.request`）

- 多次 `food.analysis.request` + `food.analysis.confirm.request` 之后
- 设备发起一次 `nutrition.analysis.request` 完成本餐汇总并入库

---

## 6. 多语言与单位

- 协议请求必须携带 `meta.locale`，并透传至视觉调用与营养查询。
- vision prompt 模板按 `meta.locale` 切换，确保返回与设备 UI 一致的语言。
- 服务端重量统一 `g`，设备 UI 可本地化展示 `oz/g`。
- 营养值字段命名英文统一（`calories / protein / fat / carbs / fiber / sodium ...`），多语言只在展示层做。

---

## 7. 质量与风控

| 维度 | 策略 |
| --- | --- |
| 超时 | 任一 Provider 超时（默认 1.5s）按 miss 处理，立即跳到下一个来源 |
| 失败 | 三连失败的 Provider 短期熔断（区域内独立计数） |
| 置信度 | `llm_estimated` 强制 `confidence < 0.5` |
| 可追溯 | 所有 `meal_item` 必须保留首次命中的 `source` 与 `providerRequestId` |
| 监控 | 上线后持续观察：`source` 分布、确认率、修正率、命中率、超时率 |
| 合规 | 国内调用不出境；国外调用不回国；`user_common` 不跨区同步 |

---

## 8. 实施清单（建议顺序）

1. 视觉 provider 由 `env` 驱动（区域 + 首选 + 兜底），prompt 模板强制输出 `count`。
2. `items[]` 协议改造（替换 `recognitions[]`）+ `source` 枚举对齐。
3. 新增 `food.analysis.confirm.request/result` 链路与 `user_common` 写入。
4. 营养回查路由按 §4.1 / §4.2 固化，区域配置完全隔离。
5. CN：接入 `boohee` 与 `cn_internal`；US：接入 `usda_fdc` + `edamam`，验证 `open_food_facts` / `nutritionix` 子集。
6. 汇总链路（`nutrition.analysis.request`）落库与统计指标接入。

---

## 9. 最终结论

阶段一研发落地口径：

```text
视觉（含 count）必须可配置
+ user_common 优先沉淀
+ 国内/国外两条独立 Provider 链
+ LLM 营养兜底（低置信标记）
+ 设备确认后写库
+ 本餐结束统一汇总
+ 多语言 locale 透传
```

国内链路落地的关键拼图：`Qwen-VL + boohee + cn_internal + LLM 兜底`。  
国外链路落地的关键拼图：`OpenAI Vision + USDA + Edamam + LLM 兜底`。  

两条链路均坚持「AI 给候选、用户做决定、设备秤定重量、来源全可追溯」的硬约束，是当前最稳妥的 MVP 技术路径，也为后续接入 FatSecret / Passio 等更重的商业方案保留了可替换接口。
