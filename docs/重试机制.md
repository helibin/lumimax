# 重试机制总览

> 状态：与当前代码一致（2026-05）。  
> 适用范围：Lumimax 后端（gateway / biz-service / iot-service / base-service）及设备侧约定。  
> 关联：[IoT 通讯模块规范](./IoT通讯模块规范.md)、[api/docs/RabbitMQ消息规范.md](../api/docs/RabbitMQ消息规范.md)、[项目架构总览与开发约束](./项目架构总览与开发约束.md)。

本文说明：**哪里会自动重试**、**哪里只幂等不去重试**、**哪里根本没有重试**。排查问题时先对号入座，避免在「不会重试」的链路上等设备或 MQ 自动恢复。

## 1. 速查表

| 链路 | 是否自动重试 | 实现位置 | 说明 |
|------|-------------|----------|------|
| IoT **下行**（biz → RMQ → iot → EMQX/云） | ✅ 有 | `iot-downstream.rabbitmq.controller.ts` | 应用层退避 + `nack(requeue=true)`，最多 9 档 |
| IoT **上行** bridge（MQTT 共享订阅 -> RMQ publish） | ✅ 部分有 | `emqx-provider.service.ts`、`iot-bridge.publisher.service.ts` | MQTT 连接自动重连；写 RMQ 必须配 publisher confirm |
| IoT **上行** biz 消费 | ❌ 应用层无退避 | `iot-biz-events.rabbitmq.controller.ts` | 同上 |
| IoT **上行** 幂等 | ✅ 去重（非重试） | `iot-message-registry.service.ts`、`iot-ingest.service.ts` | 同 `requestId` 重复投递直接 ack 跳过 |
| EMQX **鉴权 / ACL** | ❌ 无 | `emqx-ingress.service.ts`、`device-access-validation.service.ts` | 每次请求单次判定 |
| 服务端 **MQTT 客户端** 连 EMQX | ✅ 库级重连 | `emqx-provider.service.ts` | `reconnectPeriod: 5000ms` |
| 饮食 **第三方 HTTP** | ❌ 无循环重试 | `provider-http.client.ts` | 仅标记 `retryable`；换 Provider 属 fallback |
| 饮食 **用户重识别** | 业务语义 | `diet-application.service.ts` | `retry_recognition_selected`，非基础设施重试 |
| 设备 **OTA** | 设备侧 | `device-application.service.ts` 下行 payload | 字段 `data.retry`，默认 3 |
| **对象存储**上传 | 幂等复用 | `base-service` `storage.service.ts` | 同 `requestId` 复用 objectKey |
| Gateway **限流** | 客户端退避提示 | `gateway-rate-limit.service.ts` | 响应 `retry-after` |
| AWS IoT **SQS** 消费 | 依赖 SQS 可见性 | `aws-iot-sqs.consumer.ts` | 失败不 ack，由 SQS 再次投递 |

## 2. IoT 下行（唯一完整的应用层退避重试）

### 2.1 链路

```text
biz-service  IotDownlinkService.publish()
  → 写入 iot_messages（outbox，direction=downstream）
  → RabbitMQ routing key: iot.down.publish
  → 队列 lumimax.q.iot.stream（默认，见 api/configs）
  → iot-service  IotDownstreamRabbitmqController
  → iot-kit 向 EMQX HTTP API / 云 IoT 发布
  → 设备 MQTTS 订阅
```

biz-service **不**直接连 EMQX 发布；重试逻辑只在 **iot-service** 消费下行队列时执行。

### 2.2 代码与退避表

| 项 | 值 |
|----|-----|
| 控制器 | `api/apps/iot-service/src/transport/iot-downstream.rabbitmq.controller.ts` |
| Outbox / `retry_count` | `api/apps/iot-service/src/transport/iot-downlink.service.ts`（biz 侧镜像：`api/apps/biz-service/src/iot/transport/iot-downlink.service.ts`） |
| 表字段 | `iot_messages.retry_count` |
| 退避间隔（ms） | `1000, 1000, 5000, 10000, 15000, 30000, 30000, 60000, 300000`（共 9 档） |
| 最大轮次 | `DOWNSTREAM_MAX_BACKOFF_ROUNDS = 9` |

失败处理顺序：

1. 非永久错误 → `markFailedById(..., 'failed')`，`retry_count += 1`
2. 按当前 `retry_count` 查下一档 `delayMs`
3. `await delay(delayMs)` 后 `nack(message, requeue: true)`，消息回到队列
4. 超过 9 档 → `requeue: false`，outbox 保持 `failed`（或达上限后不再 requeue）

**永久错误**（`PermanentDownstreamError`，不再 requeue，outbox → `dead`）示例：

- `device not found`
- `messageId` / `deviceId` / `topic` 缺失
- payload `direction` 非 `down`
- **topic 内 deviceId 与 payload.deviceId 不一致**（与设备 ACL `topic_device_id_mismatch` 同类约束）
- EMQX REST 未配置 `IOT_ACCESS_KEY_ID` / `IOT_ACCESS_KEY_SECRET`

### 2.3 终态与重复消费

已 `published` / `dead` 的消息再次投递时，`skipAlreadyFinalized` 直接 **ack**，不重复发布。

下行 `messageKey` 由 `requestId + topic` 派生；重复 `requestId` 建 outbox 时走唯一键冲突后复用已有行。

## 3. IoT 上行（共享订阅 + 幂等）

### 3.1 iot-service bridge

| 文件 | 成功 | 失败 |
|------|------|------|
| `api/apps/iot-service/src/transport/iot-bridge.rabbitmq.controller.ts` | `ack` | `nack(..., requeue: false)` |

消费 EMQX MQTT 共享订阅上行事件：归一化 → `iot_message_registry.recordReceived` → 发布 `biz.iot.message.received`。

重复上行（`recorded.duplicate` 且状态已在 `received|queued|processing|handled|skipped`）→ **ack 跳过**，不算失败重试。

### 3.2 biz-service 业务消费

| 文件 | 成功 | 失败 |
|------|------|------|
| `api/apps/biz-service/src/iot/transport/iot-biz-events.rabbitmq.controller.ts` | `ack` | `nack(..., requeue: false)` |

`IotIngestService.ingestNormalizedMessage` 内对 `requestId` 做业务幂等；重复请求返回 `duplicate skipped`，不向设备重复下发成功结果。

### 3.3 与文档的差异

MQTT 上行链路不依赖 EMQX 直接投 RabbitMQ，因此这里的关键可靠性不再是「上行 RMQ consumer 退避」，而是：

- `iot-service` MQTT client 自动重连
- 上行写 RabbitMQ 时启用 publisher confirm
- `deviceId + requestId` 幂等去重
- bridge / biz 消费失败进入 DLQ

## 4. EMQX 鉴权 / ACL（无重试）

设备每次 `connect` / `publish` / `subscribe` 触发 HTTP 回调：

| 阶段 | 服务 | 文件 |
|------|------|------|
| 鉴权 | iot-service | `ingress/emqx-ingress.service.ts` → `authenticate` |
| ACL | iot-service | `authorize` → `DeviceAccessValidationService.validateTopicAccess` |

判定规则（节选）：

- `clientId` 必须对应库内设备主键
- Topic 中 `{deviceId}` 必须等于认证得到的 `deviceId`
- 否则 `topic_device_id_mismatch`（`40310` / `iot.topic_forbidden`）

**每次 publish 独立判定，失败不会自动重试。** 设备侧需修正 `clientId`、证书与 topic 模板后重新发送。参见 [设备协议模块规范](./设备协议模块规范.md) §3.3。

## 5. MQTT 连接层重连（非业务重试）

`api/packages/iot-kit/src/providers/emqx/emqx-provider.service.ts`：

- `reconnectPeriod: 5000`（mqtt.js 自动重连）
- `connectTimeout: 15000`、`INITIAL_CONNECT_WAIT_MS: 20000`

既包括下行 MQTT publish，也包括 **iot-service 作为 MQTT 客户端连 EMQX 并维持共享订阅**；它不是业务级重试，但它是上行链路可恢复性的前提。

## 6. 饮食中心（Food Query / Provider）

### 6.1 HTTP 客户端

`api/apps/biz-service/src/diet/providers/provider-http.client.ts`：

- 对超时、网络错误、429、5xx 等设置 `retryable: true`（`provider-error.ts`）
- **当前没有**「同一 URL 自动重试 N 次」的循环；一次请求失败即抛错

### 6.2 Provider 链 fallback（不是重试）

`FoodQueryRouter` 按 `provider-routes.yaml` 顺序尝试多个 Provider；某一源 miss/失败会换下一个源，属于 **路由 fallback**，不是对同一 Provider 的退避重试。

### 6.3 用户「重新识别」

设备/App 选择 `retry_recognition_selected`（或 optionId `retry:...`）时，走 **新的确认/查询路径**，与 MQ 或 ACL 重试无关。

## 7. 设备指令与其它

### 7.1 OTA 下行

`device-application.service.ts` 在 `cmd.ota.upgrade` 的 `data` 中带 `retry`（默认 **3**），由 **设备固件** 在 OTA 失败时重试，不是服务端 RMQ 重试。

### 7.2 对象存储

`base-service` `storage.service.ts`：相同 `requestId` 生成相同 `objectKey`，客户端重传同一请求时复用记录（幂等），不是失败自动重试。

### 7.3 Gateway 限流

超限返回 `42901` / `request.too_many`，响应头含 **`retry-after`**（毫秒），由调用方退避后重试。

### 7.4 同步 HTTP API（gateway → biz）

架构约定（见 [emqx-middle-plan/02-three-service-architecture-and-sequences.md](./emqx-middle-plan/02-three-service-architecture-and-sequences.md) §7.1）：

- 创建类请求 **禁止** 网关/客户端盲重试
- 依赖 `requestId` / 幂等键由服务端去重

## 8. AWS IoT（可选部署）

`api/apps/iot-service/src/ingress/aws-iot-sqs.consumer.ts`：

- 处理成功 → SQS 消息删除（ack）
- 处理失败 → 日志 `retry: true, acked: false`，消息在 **可见性超时** 后由 SQS 再次投递
- 无应用内 1s/5s/30s 退避表

## 9. 运维与排查

### 9.1 查下行重试次数

```sql
SELECT id, message_key, device_id, topic, status, retry_count, error_json, updated_at
FROM iot_messages
WHERE direction = 'downstream'
ORDER BY updated_at DESC
LIMIT 20;
```

### 9.2 日志关键字

| 关键字 | 含义 |
|--------|------|
| `IoT 下行消息发布失败，准备第` | 下行退避重试中 |
| `已达退避重试上限` | 9 档用尽 |
| `IoT 下行消息已标记为 dead` | 永久错误 |
| `EMQX ACL 已拒绝` | ACL 单次拒绝，需修设备 topic/clientId |
| `跳过重复上行消息` | 幂等命中，非失败重试 |
| `业务侧消费 IoT 业务事件失败` | biz 消费失败，nack 且不 requeue |

### 9.3 常见问题

| 现象 | 是否有自动重试 | 建议 |
|------|----------------|------|
| `topic_device_id_mismatch` | ❌ | 对齐 `clientId` 与 topic 中 deviceId |
| 下行 publish 超时 / 5xx | ✅（最多 9 档） | 查 `iot_messages.retry_count`、EMQX API |
| 上行业务处理抛错 | ❌（代码层） | 查 biz 日志；配置 DLQ 时看死信队列 |
| EMQX 与 iot-service MQTT 断连 | ✅（连接层） | 查 `reconnectPeriod`、broker 连接日志、共享订阅恢复情况 |
| 第三方营养 API 超时 | ❌（单请求） | 看是否已 fallback 到下一 Provider |

## 11. 文档与实现差异（维护时注意）

| 文档描述 | 当前代码 |
|----------|----------|
| RMQ 消费失败 3 次、1s/5s/30s | **仅下行** iot-service 实现退避，且为 **9 档** 另一套间隔；上行主链路已改为 MQTT 共享订阅 |
| 饮食 `provider-http`「指数退避」 | 仅有 `retryable` 标记，**无**自动重试循环 |
| `retry.event` / `diet.food.analyze.retry.queue` | 规范中标注为**可选**，需对照部署是否启用 |
