## 2. 阶段一关键决策（本期硬约束）

以下决策已经覆盖了 Doc-MVP 中的部分原文表述，开发以本节为准。

### 2.1 设备身份与「无主用户态」

- 设备端**只有一个匿名 principal**与云端 IoT 通讯；存在概念上的 **「无主用户态」**：设备初始化后先以临时/匿名身份工作，后续可升级到与用户账号体系关联。
- 阶段一：设备身份即业务主键；**数据沉淀维度以 `device_id` 作为 `user_common` 等表的逻辑 user 标识**（与饮食策略文档一致）。
- 阶段二：设备与用户为 **多对多** 关系（一台设备多用户、一用户多设备）；迁移时以 `device_id` 为 join key 归户。
- 不引入 Cognito、不做 1 对 1 强制绑定（阶段一）。
- 业务表 **预留 `user_id` 字段**（默认空），阶段二回填。

### 2.2 IoT 接入与消息总线

- **EMQ X 自建为主**，公有云 IoT（已对接 **AWS** 等）**设计预留**为容灾或局部辅助；本期无设备端实时称重流直连业务服务。
- **RabbitMQ**：在 **IoT bridge** 将 MQTT 业务事件 **转换为 RabbitMQ 消息** 后，再分发给 **biz-service**（及其他消费者），避免 biz 直接绑 MQTT 耦合。
- 当前统一采用 [设备接入协议规范v1.3.md](设备接入协议规范v1.3.md) 作为联调与实现基线。
- topic / envelope 风格继续保留 `meta.event` + `data` 模式。

### 2.3 饮食中心

- **国内 + 国外双部署**，单部署只服务一个区域；区域由 `DIET_DEPLOYMENT_REGION` 决定。
- 视觉模型由 `DIET_VISION_*` env 选择（cn / us 各一）。
- **视觉（LLM / 多模态）**：本期必须输出 **名称、可信度、数量（count）**；数量以 **prompt 工程** 为主，视觉模型能解析则填 `count`，否则确认页由用户补全。
- 营养来源**回查顺序**（与 [food-provider-strategy.md](food-provider-strategy.md) §5 一致）：

  ```text
  1. user_common（用户沉淀；阶段一以 device_id 为维度）
  2. third_party（按区域与配置选子集，顺序可配置）：open_food_facts / usda / edamam / boohee / nutritionix
  3. llm_estimated（营养兜底；OpenAI / Gemini / Qwen / DeepSeek 等均为可选 provider，由环境变量选择其一）
  ```

- **国内 cn**：第三方接 **薄荷健康（boohee）**；仍只识别**食物原材料**，不做成品菜拆解。
- 单次 `food.analysis.request` 返回 **种类 + 数量（或 null）+ 重量 + 预估营养**（便于设备 UI 展示热量等）；用户确认后**写入 `user_common`**。
- **本餐结束才进行最终营养汇总**（`nutrition.analysis.request`）；本期不做 Doc-MVP §S7 的实时差值动画。

### 2.4 单位与计量

- 端云所有 API 数据**强制公制 g**。
- 设备端 UI 可在 g / oz 之间切换显示，转换仅在端上完成。

### 2.5 饮食计划

- 设备端单选**预置模板**（如 `FAT_LOSS_1500`、`MUSCLE_2500`）。
- 不做宏量营养素自由输入引擎。

### 2.6 多语言（i18n）

- **声明**：需根据 **设备端或用户端设置的语言** 做多语言适配（文案、错误提示、食物展示名等）。
- 协议层：在 `meta` 中携带 **`locale`**（如 `en-US`、`zh-CN`），服务端与第三方调用时透传；视觉与营养 LLM 的 prompt 随 `locale` 切换。
- 管理后台与官网：与设备/App 同一套语言资源策略（详见前端仓库约定）。

### 2.7 不做清单（本期）

| 不做的能力 | 归属阶段 |
| --- | --- |
| 家庭群、主从、监护人、扫码登录、临时账号（完整 Doc-Auth） | 阶段二（Doc-Auth） |
| Cognito / 第三方账号登录 | 阶段二 |
| Kafka + Timescale + 独立 Weight Service | 长期 |
| 训练闭环、Model OTA、标注平台 | 长期 |
| intervention 独立模块、S7 实时差值动画 | 下一期再定（见已拍板 §7） |
| 多品牌/多租户 | 长期 |
| 设备实时干预动画（Doc-MVP §S7 完整版） | 阶段二或后续 |
| Nutritionix **强依赖**（us 无 Key 不阻塞） | 保持弱依赖 |

### 2.8 技术基线（仓库级）

- **后端**：monorepo + **pnpm** + **NestJS**；三微服务 **gateway / base-service / biz-service**；数据库 **PostgreSQL + Redis**；服务间异步 **RabbitMQ**（先简单 exchange/queue 命名，再演进）。
- **前端**：同 monorepo + pnpm；**后台管理（admin）** 与 **官网（web）**；**共用一套 admin 设计系统**；admin **本期做 RBAC**。

## 5. 饮食 MVP 工作流（PR 切片顺序）

> 本节是「同一个改动如何拆 PR」的说明，与 [api/AGENTS.md](../AGENTS.md) 的服务边界规则配合使用。

### 5.1 PR 切片（推荐顺序）

| # | PR 标题 | 主要改动 | 依赖 |
| --- | --- | --- | --- |
| 1 | `feat(diet): vision provider env-driven` | `VisionProviderFactory` 改为 env 选择；新增 `DIET_VISION_*` 配置；移除 region 参数 | — |
| 2 | `feat(diet): add cn vision provider (qwen-vl)` | 在 `providers/vision/` 新增国内视觉 provider 并注册 | PR 1 |
| 3 | `refactor(diet): nutrition router v2` | 路由：`user_common → third_party（OFF/USDA/Edamam/boohee/Nutritionix 可配置）→ llm_estimated`；`DIET_NUTRITION_ESTIMATOR_*` 选兜底模型 | PR 1 |
| 4 | `feat(diet): seed cn food knowledge` | 国内自建食品库种子数据 + 灌库脚本 | 独立 |
| 5 | `feat(diet): soft-timeout vision and nutrition providers` | provider 层软超时不抛错，落到下一来源 | PR 1, 3 |
| 6 | `feat(diet): finish-meal nutrition summary` | `finish_meal` 汇总营养 + 写 `user_common`；单次回包含 **预估营养** | PR 3 |
| 7 | `docs(diet): rewrite food-provider-strategy + roadmap cross-link` | 文档同步（已部分完成） | 独立 |

### 5.2 切片原则

- 按 reviewer 边界拆分 PR（见 §5.1）；先 factory 后实现；先 IoT bridge/RabbitMQ 契约再 biz 消费方。
- 严禁 `git add .`；每 PR 输出修改文件列表 + 验证命令（与 [AGENTS.md](../AGENTS.md) §「输出要求」一致）。
- 跨服务（gateway / base / biz）的契约变更必须先合 proto/DTO，再合调用方。
- RabbitMQ 消息格式与路由键在 `api/packages/messaging/`（或项目约定路径）统一封装后再被各服务引用。

### 5.3 验证最小集

- biz `food-analysis.service.spec.ts` 用例覆盖：vision miss → user_common miss → 3rd hit。
- e2e（gateway）跑一次 `food.analysis.request` + `nutrition.analysis.request`。
- cn / us 各跑一次 smoke（CI matrix env，可后置）。
