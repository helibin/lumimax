## 1. 模块定位

设备协议规范定义"**设备和云之间发什么、收什么、字段叫什么、状态怎么走**"。它是：

- **设备固件**的实现合同
- **云端 IoT 接入层**（`apps/iot-service/` 传输 + `apps/biz-service/src/iot/` 业务 ingest）的解析合同
- **饮食中心**（`apps/biz-service/src/diet/`）和**设备域**（`apps/biz-service/src/device/`）的输入输出合同

它**不**定义：

- broker（EMQ X / AWS IoT）配置和部署 → 见《IoT 通讯模块规范》
- 证书生命周期、ACL → 见《IoT 通讯模块规范》§3 + EMQX 中等方案
- gateway HTTP 路由 → 见 §§9

## 2. 总体决策（v1.3 对 v1.2 的升级，已生效）

| 维度 | v1.2 | v1.3 |
| --- | --- | --- |
| 设备身份 | 隐含与用户绑定 | **匿名设备身份**，`meta` 不传 `principalType` |
| Broker | 未明确 | **EMQ X 自建**为主，公有云为容灾预留 |
| 消息总线 | 未定义 | IoT bridge 将 MQTT 转为 **RabbitMQ**，再分发 biz |
| meal 标识 | `mealRecordId` 或别名 | **只有** `mealRecordId` |
| 识别结果 | `recognitions[]` | **改为** `items[]` |
| 用户确认 | 无独立事件 | 新增 `food.analysis.confirm.request/result` |
| 语言/市场 | 混用 `locale` | **`locale`** 只做国际化；**`market`** 由设备属性决定 |
| 饮食计划 | 可扩展 | **移除** `attr.dietPlan.set` |

⚠️ 接下来如要破坏字段（重命名、改类型、移除必填），**必须升 v1.4** 并走文档评审；不允许在 v1.3 上 silent break。

## 3. 设备身份

### 3.1 业务身份

- **`deviceId`** 是设备在云端的唯一业务主键。
- 由后台 `POST /admin/devices` 创建时分配（小写 ULID，约 26 位），格式：`^[a-z0-9][a-z0-9_-]{0,127}$`，长度 ≤ 128。
- 设备出厂时烧录 `deviceId`，固件不可改。

### 3.2 协议身份（envelope.meta）

```jsonc
{
  "meta": {
    "requestId": "01jvq4w0e29b41d4a716446651",  // 32 位小写 uuid32 / ULID（设备生成）
    "deviceId":  "01kqaey25yg7cdqc699vr74e6r",  // 等于业务 deviceId
    "timestamp": 1710000010000,                  // ms (UTC)
    "event":     "food.analysis.request",        // 见 §5
    "version":   "1.3",                          // 业务协议版本
    "locale":    "en-US"                         // 可选；只影响 i18n
  },
  "data": { /* event 专属 */ }
}
```

字段约束（见 `api/internal/contracts/src/iot/schemas/envelope.schema.json`）：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `requestId` | ✅ | 每次设备发起唯一；重发使用同一个；服务端按此做幂等去重 |
| `deviceId` | ✅ | 必须等于 MQTT clientId 和证书中解析出的 deviceId |
| `timestamp` | ✅ | 毫秒 Unix 时间戳；UTC |
| `event` | ✅ | 见 §5 事件枚举 |
| `version` | ✅ | 固定 `1.3` |
| `locale` | ❌ | 仅多语言透传；BCP-47 形如 `en-US`、`zh-CN` |

### 3.3 三个身份必须一致

```text
MQTT clientId == meta.deviceId == 证书 CN/SAN 解析出的 deviceId
```

任意一个不等于另外两个 → **拒绝连接 / 拒绝消息**。

### 3.4 不再使用的字段

- `meta.principalType`：**移除**（v1.3）
- `meta.userId`：阶段一不传；阶段二开启账号体系时再加，届时**新增字段不破坏**。
- `data.market`：**设备不要主动传**（v1.3）。服务端取 `device.market`，再回退到 `mealRecord.market`，最后回退到 `DEFAULT_MARKET`。

## 4. Topic 命名（业务协议层）

主基线 v1.3 使用的 topic 模板：

```text
v1/connect/{deviceId}/req | res
v1/status/{deviceId}/req  | res
v1/event/{deviceId}/req   | res
v1/attr/{deviceId}/req    | res
v1/cmd/{deviceId}/req     | res
```

> ⚠️ EMQX 中等方案推荐的 topic 命名空间是 `lmx/v1/devices/{deviceId}/up|down/...`（见 [`docs/emqx-middle-plan/01-protocol-and-topic-contract.md`](emqx-middle-plan/01-protocol-and-topic-contract.md) §4）。
> 当前实现仍使用 `v1/{category}/{deviceId}/{direction}`（`TopicParserService.parse` 强校验 4 段 + version=`v1`）。
> 切换到 EMQX 命名空间属于**协议小升级**：要么走 v1.4 兼容期，要么由 broker 侧做 topic rewrite，**不在本文件单方面变更**。

### 4.1 当前实现支持的 topic kind

由 `api/apps/biz-service/src/iot/iot.types.ts`（及 iot-service 侧同名类型）`BizIotTopicKind` 枚举强约束：

| Kind | category / direction | 用途 |
| --- | --- | --- |
| `CONNECT_REQ` | connect / req | 设备激活 / 注册 |
| `CONNECT_RES` | connect / res | 注册结果 |
| `STATUS_REQ` | status / req | 心跳、状态 |
| `EVENT_REQ` | event / req | 业务上行（餐次、识别、上传凭证申请等） |
| `EVENT_RES` | event / res | 业务下行响应 |
| `ATTR_REQ` | attr / req | 属性应用回执 |
| `ATTR_RES` | attr / res | 属性下发 |
| `CMD_REQ` | cmd / req | 指令执行回执 |
| `CMD_RES` | cmd / res | 指令下发 |

### 4.2 ACL（broker 强制）

设备 `{deviceId}` 只允许：

- **publish**：`v1/{category}/{deviceId}/req`（自己的 req 通道）
- **subscribe**：`v1/{category}/{deviceId}/res`（自己的 res 通道）

任何对其他设备 topic 的发布/订阅都必须被 broker 拒绝；ACL 在《IoT 通讯模块规范》§4 落地。

## 5. 事件总览

### 5.1 上行（设备 → 云）

| Event | Topic | 说明 |
| --- | --- | --- |
| `connect.register` | `v1/connect/{deviceId}/req` | 设备注册激活 |
| `status.heartbeat` | `v1/status/{deviceId}/req` | 心跳、电量、Wi-Fi RSSI |
| `upload.token.request` | `v1/event/{deviceId}/req` | 图片上传凭证申请 |
| `upload.url.request` | `v1/event/{deviceId}/req` | 图片上传预签名 URL 申请 |
| `meal.record.create` | `v1/event/{deviceId}/req` | 创建本餐 |
| `food.analysis.request` | `v1/event/{deviceId}/req` | 食物识别（图 + 重量） |
| `food.analysis.confirm.request` | `v1/event/{deviceId}/req` | 用户确认 / 修正（v1.3 新增） |
| `nutrition.analysis.request` | `v1/event/{deviceId}/req` | 结束本餐汇总 |
| `attr.set.result` | `v1/attr/{deviceId}/req` | 属性下发回执 |
| `cmd.*.result` | `v1/cmd/{deviceId}/req` | 指令执行回执 |

### 5.2 下行（云 → 设备）

| Event | Topic | 说明 |
| --- | --- | --- |
| `connect.register.result` | `v1/connect/{deviceId}/res` | 注册结果 |
| `status.heartbeat.result` | `v1/status/{deviceId}/res` | 心跳回执（可选） |
| `upload.token.result` | `v1/event/{deviceId}/res` | STS 临时上传凭证结果 |
| `upload.url.result` | `v1/event/{deviceId}/res` | 上传预签名 URL 结果 |
| `meal.record.result` | `v1/event/{deviceId}/res` | 本餐创建结果 |
| `food.analysis.result` | `v1/event/{deviceId}/res` | 识别候选（含 `items[]`） |
| `food.analysis.confirm.result` | `v1/event/{deviceId}/res` | 用户确认结果（v1.3 新增） |
| `nutrition.analysis.result` | `v1/event/{deviceId}/res` | 本餐汇总结果 |
| `attr.set` | `v1/attr/{deviceId}/res` | 参数下发 |
| `cmd.*` | `v1/cmd/{deviceId}/res` | 指令下发 |

### 5.3 事件名命名规范（新增事件时遵守）

```text
<domain>.<entity>.<action>[.result]
```

- 全小写，点分。
- 上行**请求**：`xxx.request`；上行**回执/结果**：`xxx.result`。
- 下行**指令**：`cmd.<domain>.<action>`；下行**响应**：`xxx.result`。
- 同一对 req/res 必须能通过 `meta.requestId` 一一对应（响应回写相同 requestId 或在 `meta.correlationId` 写来源 requestId，**目前主线使用 requestId 直接对照**）。

## 6. connect / status

### 6.1 `connect.register`（上行）

```json
{
  "meta": {
    "requestId": "01jvq0w8ak4n9p6r2t7x3y5bcd",
    "deviceId":  "01kqaey25yg7cdqc699vr74e6r",
    "timestamp": 1710000000000,
    "event":     "connect.register",
    "version":   "1.3",
    "locale":    "en-US"
  },
  "data": {
    "firmwareVersion": "1.0.3",
    "hardwareVersion": "scale-v1",
    "protocolVersion": "1.0",      // 设备端实现的子协议版本（非业务版本）
    "network":         "wifi"
  }
}
```

**服务端行为**：

- 写 `device.last_seen_at` / 更新 `firmware_version` / 设置 `status=active`。
- 不接收 / 不信任 `data.country` 等地理字段；`market` 不由设备主动上报。
- `connect.register` 用于设备激活与 `locale` 更新。
- 同 requestId 重发幂等：返回同一逻辑结果。

### 6.2 `connect.register.result`（下行）

```json
{
  "meta": { "...": "...", "event": "connect.register.result" },
  "data": { "code": 0, "msg": "ok", "status": "active", "locale": "en-US", "market": "US" }
}
```

`data.market` 是服务端权威值，设备应以此为准（仅用于显示，业务不影响）。

### 6.3 `status.heartbeat`

```json
{ "data": { "battery": 78, "wifiRssi": -55 } }
```

`status.heartbeat.result` 可选；不返回不影响心跳节奏。

## 7. STS 上传凭证 / 预签名 URL

### 7.1 `upload.token.request`（上行）

`upload.token.request` 走 STS 临时凭证模式。

```json
{
  "meta": { "...": "...", "event": "upload.token.request" },
  "data": { "filename": "food-001.jpg", "fileType": "image/jpeg", "maxBytes": 5242880 }
}
```

### 7.2 `upload.token.result`（下行）

```json
{
  "meta": { "...": "...", "event": "upload.token.result" },
  "data": {
    "code": 0,
    "msg": "ok",
    "provider": "aws",
    "mode": "credentials",
    "uploadMode": "sts_credentials",
    "bucket": "lumimax-assets",
    "region": "ap-southeast-1",
    "objectKey": "tmp-file/device/{deviceId}/upload_01jvq0w5zk8m6n3p7r4t9x2yab.jpg",
    "tmpPrefix": "tmp-file/device/{deviceId}/",
    "credentials": {
      "accessKeyId": "ASIA...",
      "secretAccessKey": "...",
      "sessionToken": "..."
    },
    "expiresAt": 1710000300000,
    "maxBytes": 5242880
  }
}
```

- objectKey 必须落在设备 prefix：`tmp-file/device/{deviceId}/`（详见总览 §8）。
- `upload.token.result` 不返回 `uploadUrl`、`method`、`headers`，这些只属于 `upload.url.result`。
- 设备使用 `credentials` 直传对象存储。
- 临时凭证有效期以 `expiresAt` 为准；上传范围限制在 `tmpPrefix` 下。

### 7.3 `upload.url.request`（上行）

`upload.url.request` 只申请一次性预签名上传 URL，服务端会强制按 `presigned-url` 模式处理。

```json
{
  "meta": { "...": "...", "event": "upload.url.request" },
  "data": { "filename": "food-001.jpg", "fileType": "image/jpeg", "maxBytes": 5242880 }
}
```

### 7.4 `upload.url.result`（下行）

```json
{
  "meta": { "...": "...", "event": "upload.url.result" },
  "data": {
    "code": 0,
    "msg": "ok",
    "provider": "aws",
    "uploadMode": "presigned_put",
    "bucket": "lumimax-assets",
    "region": "ap-southeast-1",
    "objectKey": "tmp-file/device/{deviceId}/upload_01jvq0w5zk8m6n3p7r4t9x2yac.jpg",
    "uploadUrl": "https://...signed-url...",
    "method": "PUT",
    "headers": { "Content-Type": "image/jpeg" },
    "expiresAt": 1710000300000,
    "maxBytes": 5242880
  }
}
```

- 设备必须使用返回的 `method + uploadUrl + headers` 上传文件。
- 后续 `food.analysis.request.data.imageKey` 使用返回的 `objectKey`。

## 8. 餐次主链路

### 8.1 `meal.record.create` / `meal.record.result`

请求：

```json
{ "data": { "startedAt": 1710000000000 } }
```

响应：

```json
{ "data": { "code": 0, "msg": "ok", "mealRecordId": "01hx...", "status": "active", "startedAt": 1710000000000 } }
```

### 8.2 `food.analysis.request`

```json
{
  "meta": { "...": "...", "event": "food.analysis.request", "locale": "en-US" },
  "data": {
    "mealRecordId":  "01hx...",
    "weight":        123.5,
    "unit":          "g",
    "imageKey":      "tmp-file/device/.../food-001.jpg",
    "measuredAt":    1710000010000
  }
}
```

约束：

- `weight` 单位**必须** `g`（数值 ≥ 0.1）；端上 UI 切换 `g/oz` 不在协议里。
- `imageKey` 可空（手输/语音场景）；非空时必须属于设备 token prefix（服务端校验）。
- `measuredAt` 是设备称重时刻，**不**是发包时刻。
- 设备**不**应自传 `market`；服务端按 `device.market` 选 provider。

### 8.3 `food.analysis.result`

```jsonc
{
  "meta": { "...": "...", "event": "food.analysis.result" },
  "data": {
    "code": 0, "msg": "ok",
    "mealRecordId": "01hx...",
    "foodItemId":   "01hx...",         // 注意是 foodItemId，对应 HTTP body 的 itemId
    "status":       "success",
    "weight":       123.5,
    "unit":         "g",
    "items": [                         // v1.3 关键字段（取代 recognitions[]）
      {
        "foodId":              "F_USDA_BROWN_RICE",
        "name":                "Brown Rice",
        "displayName":         "Brown Rice",
        "canonicalName":       "brown_rice",
        "type":                "ingredient",  // ingredient | prepared_dish | packaged_food | restaurant_food | mixed_meal | unknown
        "quantity":            1,             // 可 null
        "measuredWeightGram":  123.5,
        "estimatedWeightGram": 123.5,
        "confidence":          0.92,
        "source":              "usda",
        "provider":            "usda_fdc",
        "verifiedLevel":       "verified",    // confirmed | verified | estimated | unverified
        "calories":            140,           // 当前重量预估
        "nutrition":           { "protein": 3.0, "fat": 1.0, "carbohydrate": 30.0, "fiber": 1.6 },
        "children":            []
      }
    ],
    "estimatedNutrition": {
      "calories": 140, "protein": 3.0, "fat": 1.0, "carbs": 30.0,
      "source": "usda", "provider": "usda_fdc", "verifiedLevel": "verified"
    },
    "requiresUserConfirmation": true,
    "confirmationOptions": [
      { "optionId": "recognized:brown_rice", "foodName": "Brown Rice", "displayName": "Brown Rice", "canonicalName": "brown_rice", "source": "recognized", "provider": "gemini", "confidence": 0.92 }
      // 固定 4 个选项见 §8.4
    ],
    "userCommonCandidates": []
  }
}
```

#### 8.3.1 `items[].source` 合法取值

```text
user_common
open_food_facts
usda
edamam
boohee
nutritionix
cn_internal
llm_estimated
vision_fallback
```

#### 8.3.2 `items[].type` 合法取值

```text
ingredient | prepared_dish | packaged_food | restaurant_food | mixed_meal | unknown
```

### 8.4 `confirmationOptions` 固定 4 选项

按饮食中心约定（[`docs/diet-service-device-protocol-v1.0.md`](diet-service-device-protocol-v1.0.md) §7.2），设备端 UI **必须**呈现 4 个固定动作：

```text
accept                  → 直接确认
chooseFromUserCommon    → 常吃类似食物
searchSystemFoods       → 系统食物库搜索
retryRecognition        → 重新识别
```

云端按 `confirmation_source` 区分用户走哪个分支（见 §8.5）。

### 8.5 `food.analysis.confirm.request`（v1.3 新增）

```json
{
  "data": {
    "mealRecordId":       "01hx...",
    "foodItemId":         "01hx...",
    "selectedFoodId":     "F_USDA_BROWN_RICE",
    "selectedFoodName":   "Brown Rice",
    "correctedName":      null,
    "correctedCount":     null,
    "correctedWeightGram": null,
    "confirmationSource": "recognized"
  }
}
```

`confirmationSource` 合法取值：

```text
recognized                       // 直接确认
user_common_selected             // 从常吃挑
system_search_selected           // 从系统库搜索挑
retry_recognition_selected       // 让云端重识别
```

服务端行为：

- 复算营养（按修正后克重 / 名称）。
- 沉淀 `user_common`（以 `device_id` 为逻辑 user）。
- 写 `recognition_logs` 一条 `confirm` 事件。

### 8.6 `food.analysis.confirm.result`

```json
{ "data": { "code": 0, "msg": "ok", "mealRecordId": "...", "foodItemId": "...", "status": "confirmed" } }
```

### 8.7 `nutrition.analysis.request`

```json
{ "data": { "mealRecordId": "01hx..." } }
```

### 8.8 `nutrition.analysis.result`

```json
{
  "data": {
    "code": 0, "msg": "ok",
    "mealRecordId":  "01hx...",
    "status":        "finished",
    "foodItemCount": 3,
    "totalCalories": 560,
    "nutritionSummary": { "protein": 22.4, "fat": 9.1, "carbohydrate": 82.5, "fiber": 6.3 },
    "containsEstimated": true,
    "finishedAt":    1710000102000
  }
}
```

`status=finished` 后再次 `analyze / confirm / finish` → 返回错误码 `40900 meal record already finished`。

## 9. attr / cmd

### 9.1 `attr.set`（下行） / `attr.set.result`（上行）

```json
// 下行
{ "data": { "attrs": { "heartbeatInterval": 60 } } }

// 上行回执
{ "data": { "code": 0, "msg": "ok", "applied": { "heartbeatInterval": 60 } } }
```

约束：

- 不允许通过 `attr.set` 下发 `dietPlan`（v1.3 已**移除**）。
- 属性的命名空间扁平（一层 key），固件按 key 应用、回 `applied` 实际生效值。

### 9.2 `cmd.<domain>.<action>`（下行） / `cmd.<domain>.<action>.result`（上行）

```json
// 下行 cmd.reboot
{ "meta": { "event": "cmd.reboot" }, "data": { "command": "reboot", "params": {} } }

// 上行回执 cmd.reboot.result
{ "meta": { "event": "cmd.reboot.result" }, "data": { "code": 0, "msg": "accepted", "status": "accepted" } }
```

阶段一已使用的 cmd 系列（可扩展，新增前先与设备团队对齐）：

- `cmd.reboot`
- `cmd.ota.upgrade` / `cmd.ota.upgrade.accepted` / `cmd.ota.upgrade.progress` / `cmd.ota.upgrade.result`

OTA 子协议详见 `api/internal/contracts/src/iot/schemas/cmd.ota.upgrade.*.schema.json`。

### 9.3 命令幂等

- 同一个命令应携带服务器生成的 `commandId`（可选 meta 字段，未来引入）。
- 设备对同 `commandId` 必须按"已执行 → 直接回 accepted/result"处理。
- v1.3 当前主线对 OTA 已有 accepted/progress/result 三段，**其它 cmd** 至少需要 `cmd.*.result`，无 accepted 阶段。

## 10. 通用规则

### 10.1 单位

- 重量统一 `g`（数值，支持小数）。
- 时间统一毫秒 unix UTC。
- 网络 RSSI 用 dBm（负数）。
- 不要在协议里用区域单位（lb / oz / kcal/oz 等）；端上 UI 自己转。

### 10.2 国际化

- `meta.locale` 透传给视觉 / 营养 LLM 的 prompt。
- 服务端缺省 `DEFAULT_LOCALE`（`en-US`）；设备未传时使用默认。
- **错误文案**：响应 `data.msg` 只放短文案（最好英文），多语言由客户端按 `data.code` 自查；目前不在设备协议里做 i18n 完整方案。

### 10.3 错误码（设备协议侧）

| code | msg | 触发 |
| --- | --- | --- |
| 0 | ok | 成功 |
| 40001 | invalid payload | 参数缺失/格式错 |
| 40400 | meal record not found | mealRecordId 不存在 |
| 40900 | meal record already finished | 餐次已结束 |
| 42200 | object key invalid | imageKey 不属于设备 prefix / 不存在 |
| 50000 | internal error | 服务端异常 |
| 50300 | analysis unavailable | 识别/营养全部不可用 |

> 设备协议错误**只**走 `data.code / data.msg`。HTTP 入口的错误码（`error.key` 等）见总览 §6.3，**不要混淆**。

### 10.4 重发与幂等

- 设备每次发起业务请求必须生成唯一 `requestId`；
- 失联补发时**使用同一个 `requestId`**，服务端按 `(deviceId, requestId, event)` 去重，TTL ≥ 24h；
- 设备不应对同一会话同时发起多个识别请求（必须收到上一条 `result` 或超时后再发）。

### 10.5 时序

- QoS 1，应用层做幂等（broker 可重投）。
- 心跳缺省 60s；服务端可通过 `attr.set.heartbeatInterval` 调；缺失 `2.5 × heartbeat` 标离线。
- 一次 `food.analysis.request` 服务端 P95 响应应在 3-5s 内（视觉 + 营养 + 兜底全部走完）。

## 11. 与 HTTP / RabbitMQ 入口的对齐

设备协议是**入口之一**。同一逻辑在 HTTP / RabbitMQ 必须能复现。

### 11.1 HTTP ↔ IoT 对齐表

| 业务动作 | IoT event | HTTP 路由 / facade operation |
| --- | --- | --- |
| 建餐 | `meal.record.create` | `POST /api/meals` → `meals.create` |
| 识别 | `food.analysis.request` | `POST /api/meals/:id/items/analyze` → `meals.items.analyze` |
| 确认 | `food.analysis.confirm.request` | `POST /api/meals/:id/items/:itemId/confirm` → `meals.items.confirm` |
| 完餐 | `nutrition.analysis.request` | `POST /api/meals/:id/finish` → `meals.finish` |
| STS 上传凭证 | `upload.token.request` | base-service/storage `storage.createTemporaryUploadCredential`（`mode=credentials`） |
| 上传预签名 URL | `upload.url.request` | base-service/storage `storage.createTemporaryUploadCredential`（`mode=presigned-url`） |

字段对齐已在《饮食中心模块规范》§7.4 列表强约束（如 `foodItemId` ↔ `itemId`）。

### 11.2 RabbitMQ envelope 转换

IoT bridge 收到 MQTT 报文后，转 RabbitMQ envelope（详见 [`api/docs/RabbitMQ消息规范.md`](../api/docs/RabbitMQ消息规范.md) §5）：

```json
{
  "meta": {
    "messageId":  "01jvq...",
    "requestId":  "<原 meta.requestId>",
    "event":      "<原 meta.event>",
    "routingKey": "iot.<domain>.<entity>.<action>.v1",
    "occurredAt": <原 meta.timestamp>,
    "deviceId":   "<原 meta.deviceId>",
    "version":    "1.0",
    "locale":     "<原 meta.locale>"
  },
  "data": { /* 原 data */ }
}
```

biz-service 消费这套 envelope，**不要**直接解析 MQTT 报文。

## 13. 兼容期与版本策略

### 13.1 两类版本号

| 版本 | 位置 | 当前值 |
| --- | --- | --- |
| 传输命名空间 | topic 第一段 `v1/...` | `v1` |
| 业务协议版本 | `meta.version` | `1.3` |

### 13.2 兼容规则

- **加字段**（向后兼容）：同 major，可在小版本里加；老固件忽略未知字段。
- **必填字段改名 / 删除**：必须升 major（`2.0`），并提供至少 1 个版本的兼容期。
- 在 v1.3 兼容期内：服务端可在 IoT bridge 做短期 `recognitions[] ↔ items[]` 等映射，**只为灰度过渡**，不作为长期方案。

### 13.3 客户端 / 设备端协商

设备 `meta.version` 与服务端实现不一致：

- minor 不一致：允许，使用 lower 的字段集。
- major 不一致：服务端可拒绝并下发 `connect.register.result.code != 0`，提示固件升级。

## 14. 联调清单（最短路径）

参见 [`api/docs/设备协议联调检查单.md`](../api/docs/设备协议联调检查单.md)。必跑：

```text
1. connect.register → connect.register.result
2. status.heartbeat → status.heartbeat.result (可选)
3. upload.token.request → upload.token.result
4. upload.url.request → upload.url.result
5. meal.record.create → meal.record.result
6. food.analysis.request → food.analysis.result (含 items[])
7. food.analysis.confirm.request → food.analysis.confirm.result
8. nutrition.analysis.request → nutrition.analysis.result
9. attr.set → attr.set.result
10. cmd.reboot → cmd.reboot.result
11. cmd.ota.upgrade → accepted → progress → result
```

每条都要检查：

- topic 是否正确（4 段 / version=v1 / deviceId 一致）
- envelope 必填字段是否齐
- `items[].source` 取值是否落在 §8.3.1 枚举内
- 修正确认路径写入 `user_food_profiles`、`recognition_logs`
- 本餐汇总的 `containsEstimated` 是否正确
- 失败场景的错误码是否走 `data.code`（不要伪装成 HTTP 状态）
