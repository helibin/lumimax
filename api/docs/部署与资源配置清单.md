# AI Scale 部署文档（架构、资源、环境、第三方账号）

## 1. 文档目标

本文用于统一 `lumimax` 项目的部署信息，覆盖：

- 项目架构与服务职责
- 服务间调用关系与关键依赖
- 部署环境要求（本地/测试/生产）
- 服务资源配置建议（CPU/内存/副本）
- 第三方账号与密钥准备清单
- 上线前检查清单

适用仓库：`/Volumes/dev/workspace/@ai/lumimax`

---

## 2. 系统架构总览

### 2.1 服务清单与职责

- `gateway`
  - 外部统一 HTTP 入口（`/api`、`/admin`）
  - 统一鉴权与请求日志
  - 文档聚合
- `base-service`
  - 用户、认证、隐私、通知后台、对象存储抽象
  - 提供 HTTP + gRPC 基础能力
- `biz-service`
  - 设备主业务域（创建设备、绑定、状态、shadow、命令）
  - 云 IoT 适配层（AWS）
  - 餐次、识别、营养分析与管理后台业务聚合

### 2.2 关键通信关系

- Client -> `gateway`：HTTP/HTTPS
- `gateway` -> 各后端：gRPC（base/biz）
- 事件总线：RabbitMQ（topic exchange）
- 状态/限流缓存：Redis
- 持久化：PostgreSQL

---

## 3. 部署环境要求

### 3.1 基础组件（必备）

- PostgreSQL 16+
- RabbitMQ 3.13+
- Redis 7+
- Node.js 22+（建议）
- pnpm 10+

### 3.2 网络与端口建议

- 服务间 gRPC 互通（内部网络）
- Gateway 暴露 80/443（或 3000）
常见服务端口（当前默认）：

- gateway: `3000`
- base-service: HTTP `3001` / gRPC `51051`
- biz-service: HTTP `3002` / gRPC `51052`

历史拆分服务已不再作为默认部署单元；若需追溯旧端口，请查阅对应阶段文档或 Git 历史。

## 4. 配置与环境变量策略

项目已采用统一规则：

- 本地开发：`configs/{env}/{service}.env` + `configs/{env}/shared.env`
- 生产环境：`NODE_ENV=production` 时**不加载 env 文件**，完全依赖部署平台注入

### 4.1 本地文件建议

- `configs/development/shared.env`
- `configs/development/gateway.env`
- `configs/development/base-service.env`
- `configs/development/biz-service.env`

参考模板：

- `configs/*.development.env.example`
- `configs/README.md`

### 4.2 安全策略

- 可以提交：`*.env.example`
- 禁止提交：真实 `configs/*.env`、任何密钥/密码
- 敏感配置仅允许放在：
  - 本地未提交文件
  - Secret Manager / K8s Secret / CI Secret Variables

## 6. 第三方账号与账密准备清单

## 6.1 通用安全项

- `JWT_SECRET`（强随机）
- 数据库账号密码（各环境隔离）
- RabbitMQ 账号密码
- Redis 鉴权（生产建议开启）

## 6.2 IoT 与外部服务

### EMQX 主线（推荐）

- `IOT_VENDOR=emqx`
- `EMQX_BROKER_URL`
- `EMQX_ROOT_CA_PEM` / `EMQX_ROOT_CA_KEY_PEM`（推荐，用于服务端签发设备 mTLS 证书）
- `EMQX_MQTT_USERNAME` / `EMQX_MQTT_PASSWORD`（可选，服务端下行发布）
- `EMQX_MQTT_CLIENT_CERT_PEM` / `EMQX_MQTT_CLIENT_KEY_PEM`（可选，服务端下行发布）

### AWS 兼容路径（IoT / S3 / STS）

- `IOT_VENDOR=aws`
- `STORAGE_ACCESS_KEY_ID` / `STORAGE_ACCESS_KEY_SECRET` / `STORAGE_REGION`
- `IOT_ACCESS_KEY_ID` / `IOT_ACCESS_KEY_SECRET` / `IOT_REGION`
- `AWS_IOT_POLICY_NAME`
- `AWS_IOT_ENDPOINT`（可选）
- `AWS_SQS_QUEUE_URL`（队列模式）
- `STORAGE_BUCKET`（`base-service` storage）
- `STORAGE_STS_ROLE_ARN`（`base-service` storage）

### Nutritionix + OpenAI（`biz-service` diet）

- `NUTRITIONIX_APP_ID`
- `NUTRITIONIX_APP_KEY`
- `LLM_VISION_AK`
- `LLM_VISION_MODEL`
- `OPENAI_TEXT_MODEL`

### 通知通道（`base-service` notification）

- SMTP：`SMTP_HOST`/`SMTP_PORT`/`SMTP_USER`/`SMTP_PASS`

## 7. 本地/测试环境部署步骤（推荐）

1. 安装依赖
  - `pnpm install`
2. 准备配置
  - 复制 `configs/*.env.example` 到 `configs/development/*.env`
  - 填入真实本地密钥
3. 启动基础设施
  - `pnpm infra:up`（或 `docker compose up -d postgres rabbitmq redis emqx`）
4. 初始化数据库
  - `pnpm db:migrate`
  - `pnpm db:seed`（如需要初始账号/权限）
5. 启动服务
  - `pnpm dev`
6. 验证
  - Gateway health/doc 路由
  - gRPC 链路联通（创建设备、认证、上传）
  - RabbitMQ 消费正常

---

## 8. 生产部署建议

- 使用 K8s 或等价平台，所有密钥走 Secret 管理
- 关闭文件加载（已由 `NODE_ENV=production` 自动实现）
- 服务最少双副本（除非明确单副本服务）
- Gateway 前置 LB + TLS 终止 + WAF（可选）
- 启用监控：
  - 服务可用性（HTTP/gRPC health）
  - RabbitMQ 堆积
  - Redis 命中/内存
  - DB 连接池与慢查询
- 日志统一采集（按 `requestId` 关联）
