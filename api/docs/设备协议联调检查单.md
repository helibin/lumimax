# 设备协议联调检查单（vNext）

> 适用文档：`设备接入协议规范v1.3.md`  
> 目标：设备端、IoT bridge、biz-service 三方按事件逐条联调并留痕。

---

## 0. 联调前准备

- [ ] EMQ X 可连通（设备证书/凭据有效）
- [ ] IoT bridge 已启用 MQTT -> RabbitMQ 转换
- [ ] biz-service 消费者已订阅对应 routing key
- [ ] 时钟同步（设备与服务端误差 < 5s）
- [ ] 所有请求携带 `requestId`、`deviceId`、`event`、`version`
- [ ] 多语言字段可用（`meta.locale`）

---

## 1. connect.register 来回

### 1.1 设备上行
- [ ] 发送 `connect.register`
- [ ] payload 含固件/硬件版本

### 1.2 服务端下行
- [ ] 收到 `connect.register.result`
- [ ] `data.code == 0`
- [ ] `status == active`

---

## 2. status.heartbeat 来回（可选回执）

### 2.1 设备上行
- [ ] 发送 `status.heartbeat`
- [ ] 上报关键状态（如电量、信号）

### 2.2 服务端回执
- [ ] 若启用回执，收到 `status.heartbeat.result`
- [ ] 回执 `code == 0`

---

## 3. upload.token.request / upload.url.request

### 3.1 upload.token.request 设备上行
- [ ] 发送 `upload.token.request`
- [ ] 包含 `filename` / `fileType` / `maxBytes`

### 3.2 upload.token.result 服务端下行
- [ ] 收到 `upload.token.result`
- [ ] 包含 `provider`、`bucket`、`region`、`objectKey`、`tmpPrefix`、`credentials`、`expiresAt`
- [ ] `mode == credentials`
- [ ] `uploadMode == sts_credentials`
- [ ] 不包含 `uploadUrl`、`method`、`headers`
- [ ] 设备可用 STS 临时凭证完成一次上传

### 3.3 upload.url.request 设备上行
- [ ] 发送 `upload.url.request`
- [ ] 包含 `filename` / `fileType` / `maxBytes`

### 3.4 upload.url.result 服务端下行
- [ ] 收到 `upload.url.result`
- [ ] 包含 `objectKey`、`uploadUrl`、`method`、`headers`、`expiresAt`、`maxBytes`
- [ ] 设备按返回的 `method + uploadUrl + headers` 完成一次上传
- [ ] 后续业务事件使用返回的 `objectKey` 作为 `target`（`type=image`）

---

## 4. meal.record.create / result

### 4.1 设备上行
- [ ] 发送 `meal.record.create`

### 4.2 服务端下行
- [ ] 收到 `meal.record.result`
- [ ] 返回 `mealRecordId`
- [ ] `status == active`

---

## 5. food.analysis.request / result

### 5.1 设备上行
- [ ] 发送 `food.analysis.request`
- [ ] 包含 `mealRecordId`、`weight`、`unit=g`、`target`
- [ ] 如需显式指定识别方式，补充 `type=image|barcode|text|voice`；默认 `image`
- [ ] `weight > 0`

### 5.2 服务端下行
- [ ] 收到 `food.analysis.result`
- [ ] `data.items` 存在且为数组
- [ ] 每个 item 至少有：`foodId/name/confidence/source/calories/nutrition`
- [ ] 若 count 能识别，`count` 为正整数；否则允许 `null`
- [ ] 本次回包为预估营养（不是最终汇总）

---

## 6. food.analysis.confirm.request / result

### 6.1 设备上行
- [ ] 发送 `food.analysis.confirm.request`
- [ ] 包含 `mealRecordId`、`foodItemId`、`selectedFoodId`
- [ ] 修正路径可选带：`correctedName` / `correctedCount` / `correctedWeightGram`

### 6.2 服务端下行
- [ ] 收到 `food.analysis.confirm.result`
- [ ] `status == confirmed`
- [ ] `code == 0`

---

## 7. nutrition.analysis.request / result

### 7.1 设备上行
- [ ] 发送 `nutrition.analysis.request`
- [ ] 包含 `mealRecordId`

### 7.2 服务端下行
- [ ] 收到 `nutrition.analysis.result`
- [ ] 包含 `foodItemCount`、`totalCalories`、`nutritionSummary`
- [ ] 若存在估算项，`containsEstimated == true`
- [ ] `status == finished`

---

## 8. attr.set / attr.set.result

### 8.1 云端下发
- [ ] 云端发送 `attr.set`

### 8.2 设备回执
- [ ] 设备回 `attr.set.result`
- [ ] `applied` 字段符合预期

---

## 9. cmd.* / cmd.*.result

### 9.1 云端下发
- [ ] 发送 `cmd.reboot`（或其他 cmd）

### 9.2 设备回执
- [ ] 收到 `cmd.reboot.result`
- [ ] `status` 为 `accepted` 或 `success`

---

## 10. 错误与重试验证

- [ ] 缺失字段时服务端返回明确错误码（非 0）
- [ ] 重复 `requestId` 具备幂等处理
- [ ] token 过期可重新申请并成功上传
- [ ] 任一事件失败不影响后续新 `requestId` 事件处理

---

## 11. 观测与日志

- [ ] 日志可按 `requestId` 追踪全链路（设备 -> IoT bridge -> RabbitMQ -> biz）
- [ ] 能统计 `items[].source` 分布
- [ ] 能统计确认率（confirm 成功率）
- [ ] 能统计 `containsEstimated` 占比
