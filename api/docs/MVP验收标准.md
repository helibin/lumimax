# Lumimax MVP 验收标准（功能改进版）

本文档用于指导 MVP 阶段功能改进的统一验收，作为产品、研发、测试、设备联调的共同标准。

## 1. 验收依据（必须同时满足）

- 交互链路基线：`docs/交互链路设计-全栈.md`
- 设备协议基线：`docs/设备接入协议规范v1.0.md`

说明：

- 如实现与上述文档冲突，以“先补文档再改实现”或“先改实现再补文档”的方式保持一致，不允许长期偏离。
- 本文档重点用于“改进功能验收”，即在不破坏当前 MVP 主链路前提下验证新增/优化内容。

## 2. 验收范围（本期 5 条主链路）

1. B 端用户登录鉴权
2. 设备与 IoT 通讯
3. 设备上报数据并请求食物分析
4. 设备请求餐食营养分析
5. 营养中心第三方对接

## 3. 全局验收门槛（所有链路通用）

### 3.1 响应与追踪

- 对外 HTTP 响应统一包含：`code`、`msg`、`data`、`requestId`
- 设备协议消息统一包含：`meta` + `data`，`meta` 至少包含 `requestId`、`deviceId`、`timestamp`、`event`、`version`
- 任一失败场景可通过 `requestId` 在网关/基础服务/业务服务日志中串联定位

### 3.2 架构与边界

- 客户端仅访问 `gateway`
- `gateway` 仅下游调用 `base-service` / `biz-service`
- `base-service` 不反向依赖 `biz-service`
- 网关只做鉴权、编排、聚合和统一错误映射，不承载领域核心逻辑

### 3.3 数据与协议一致性

- 业务主键遵循 32 位小写 ULID
- 设备 Topic 与 Event 命名遵循 `设备接入协议规范v1.0`
- 错误码与错误语义符合约定（如 `40001`、`40400`、`42200`、`50300`）

## 4. 分链路验收标准

## 4.1 链路一：B 端用户登录鉴权

对应交互链路：`POST /api/admin/auth/login`

### 必须通过

- 正确用户名密码登录成功，返回 `accessToken` 与用户资料
- 错误用户名或密码返回 401，`error.key = auth.invalid_credentials`
- 无 token 访问受保护接口返回 401，语义为未登录/无效登录态（非凭证错误语义）
- 登录成功后访问 `admin/auth/me` 能获取当前用户及权限上下文

### 失败判定

- 用户名密码错误仍返回 `user.token_invalid`
- 响应缺失 `requestId`
- 错误码正确但错误语义错位（前端无法按 `error.key` 区分场景）

### 验收证据

- 接口请求/响应样例（成功 + 失败各 1 组）
- 网关与基础服务对应 `requestId` 日志截图或日志片段

## 4.2 链路二：设备与 IoT 通讯

对应协议章节：Topic 规范、事件总览、心跳/状态、指令下发与响应、OTA

### 必须通过

- 设备上行 Topic 正确（如 `v1/status/{deviceId}/req`、`v1/event/{deviceId}/req`）
- 云端下行 Topic 正确（如 `v1/event/{deviceId}/res`、`v1/cmd/{deviceId}/res`）
- `status.heartbeat` 可稳定上报并反映在线状态变化
- 指令下发后设备有回执（`cmd.*.result` 或等价事件）
- OTA 任务可完成“下发 -> 接收 -> 进度 -> 结果”闭环（至少 1 次成功）

### 失败判定

- Topic 拼写、方向或设备 ID 占位不符合协议
- 上下行事件缺失 `meta.event` 或 `meta.version`
- 指令无回执、回执无法关联原请求 `requestId`

### 验收证据

- MQTT/IoT 消息抓包记录（至少 1 组上行 + 1 组下行）
- 设备在线状态变化记录
- OTA 全流程事件记录

## 4.3 链路三：设备上报数据并请求食物分析

对应协议事件：`food.analysis.request` / `food.analysis.result`

### 必须通过

- 设备上报 `mealRecordId`、重量信息、可选 `imageKey` 后，服务端可返回识别结果
- 成功响应包含：`mealRecordId`、`foodItemId`、`recognitions`
- 当 `imageKey` 非法或无权限时，返回协议约定错误码（如 `42200`）
- 识别请求与识别结果通过 `requestId` 一一对应

### 失败判定

- 请求成功但结果中缺少 `foodItemId` 或识别候选列表结构异常
- 图片校验失败却返回泛化错误（无法定位为对象键问题）
- 请求与结果无法关联，导致设备侧无法落状态

### 验收证据

- 成功识别样例（至少 2 个不同食物）
- 异常识别样例（非法 `imageKey`）
- 业务库落表记录（`recognition_logs`、`meal_items`）

## 4.4 链路四：设备请求餐食营养分析

对应协议事件：`nutrition.analysis.request` / `nutrition.analysis.result`

### 必须通过

- 给定有效 `mealRecordId`，可返回餐次汇总结果（如 `mealRecordId`、`foodItemCount`、`totalCalories`、`nutritionSummary`）
- 结果返回后餐次状态转为完成态（`finished` 或等价状态）
- 多次 `food.analysis.request` 累积结果能被正确汇总
- 汇总值与明细可对账（允许定义误差范围，但需固定规则）

### 失败判定

- 返回成功但餐次未完成/状态不一致
- 汇总值与明细累计差异超过约定误差且无解释
- 相同请求重复触发导致重复完餐或重复生成最终结果

### 验收证据

- 单餐多食物完整闭环样例（建餐 -> 识别多次 -> 营养汇总）
- `meal_records` 与 `meal_items` 对账截图/查询结果

## 4.5 链路五：营养中心第三方对接

对应交互链路：diet 子域营养解析；对应协议事件：营养分析相关事件

### 必须通过

- 对接至少 1 个第三方营养来源，完成“请求 -> 标准化 -> 返回”闭环
- 第三方成功时，返回结构与内部协议一致，不向设备暴露第三方差异
- 第三方超时/失败时，系统可降级（兜底数据、重试或明确错误码）
- 记录数据来源与调用结果，支持审计与问题回溯

### 失败判定

- 第三方字段直接透传，破坏统一响应结构
- 第三方抖动导致链路整体不可用且无降级
- 缺少来源标记，后续无法解释营养值来源

### 验收证据

- 第三方成功调用样例与标准化结果
- 第三方失败降级样例（超时/5xx）
- 营养来源与调用日志记录

## 5. 回归验收（改进功能必做）

- 不得破坏 `admin/dashboard/overview` 聚合字段稳定性
- 不得破坏设备基础上行（心跳、事件）与下行（指令、OTA）
- 不得破坏饮食主链路：`meal.record.create` -> `food.analysis.request` -> `nutrition.analysis.request`
- 不得改变既有错误语义键值（除非文档同步更新并评审通过）

## 6. 非功能验收（MVP 最低要求）

- 可观测性：关键链路日志可按 `requestId` 追踪
- 稳定性：核心链路连续 24 小时联调无阻断性故障
- 性能基线（建议）：核心接口 P95 延迟、失败率有监控并可回放定位
- 可运维性：第三方异常时有明确告警与降级开关

## 7. 验收执行方式（建议）

### 7.1 执行顺序

1. 先验协议一致性（Topic/Event/Payload）
2. 再验主链路功能正确性（5 条链路）
3. 最后验异常与回归

### 7.2 环境要求

- 至少一套稳定测试环境（gateway/base/biz/数据库/Redis/IoT 模拟或真实设备）
- 设备端可发送协议样例消息，或有等价模拟器脚本

### 7.3 准入/准出

- 准入：接口、Topic、事件定义冻结并有版本标识
- 准出：本文件“必须通过”项全部通过，且无 P1/P2 阻断缺陷
