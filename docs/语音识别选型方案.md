# Lumimax 语音识别服务选型方案

> 适用范围：语音输入、语音指令、饮食记录语音备注、健康饮食助手、设备交互指令。

---

## 1. 选型结论

语音识别不要直接绑定某一家云服务，建议设计为 **可插拔 Provider 架构**。

当前推荐组合：

```txt
国内 MVP：腾讯云 ASR
海外 MVP：OpenAI gpt-4o-mini-transcribe
实时增强：Deepgram / OpenAI Realtime
私有化降本：FunASR + faster-whisper
```

原因：

- 国内链路优先考虑稳定性、接入效率、微信生态和合规。
- 海外链路优先考虑多语言能力、成本和后续 LLM 联动。
- 实时对话类能力和普通语音转文字能力要拆开选型。
- 私有化和降成本能力不要放在 MVP 第一阶段，但架构上要预留。

---

## 2. 业务场景分析

Lumimax 的语音识别主要不是为了做单纯转写工具，而是服务于后续业务结构化。

典型场景：

1. 用户说：早餐吃了两个鸡蛋，一杯牛奶。
2. 用户通过语音补充饮食记录备注。
3. 设备端语音指令：开始称重、结束称重、重新识别。
4. 健康饮食助手进行语音交互。
5. 后续可能支持长音频记录、实时对话、私有化部署。

因此完整链路应为：

```txt
语音输入
  ↓
ASR 语音识别
  ↓
文本标准化
  ↓
LLM 意图识别 / 信息抽取
  ↓
结构化饮食数据
  ↓
营养分析 / 餐次记录 / 设备动作
```

示例：

用户语音：

```txt
早餐吃了两个鸡蛋，一杯牛奶，还有一片吐司
```

结构化结果：

```json
{
  "intent": "record_meal_by_voice",
  "mealType": "breakfast",
  "items": [
    { "name": "鸡蛋", "quantity": 2, "unit": "个" },
    { "name": "牛奶", "quantity": 1, "unit": "杯" },
    { "name": "吐司", "quantity": 1, "unit": "片" }
  ]
}
```

### 4.1 国内主线

|---:|---|---|

| 2 | 阿里云智能语音交互 | 如果已有阿里云 OSS、ECS、短信、日志体系，集成更顺 |
| 3 | 讯飞开放平台 | 中文和方言能力强，适合对中文识别质量要求高的场景 |
| 4 | 百度语音识别 | 中文能力不错，可作为备选 Provider |

```txt
TencentSpeechProvider
```

- 一句话识别
- 实时语音识别
- App / 小程序语音输入
- 设备短指令识别

## 6. 推荐架构设计

### 6.1 核心原则

1. 业务层不直接依赖任何云厂商 SDK。
2. 所有语音识别能力通过统一 Provider 接口接入。
3. Provider 根据区域、业务场景、成本、失败兜底策略动态选择。
4. ASR 只负责语音转文本，不负责业务意图解析。
5. 语音识别结果需要进入后置 LLM 结构化流程。

---

### 6.2 目录结构建议

```txt
apps/
└── biz-service/
    └── src/
        └── speech/
            ├── interfaces/
            │   ├── speech-provider.interface.ts
            │   └── speech.types.ts
            ├── providers/
            │   ├── tencent-speech.provider.ts
            │   ├── aliyun-speech.provider.ts
            │   ├── iflytek-speech.provider.ts
            │   ├── baidu-speech.provider.ts
            │   ├── openai-speech.provider.ts
            │   ├── deepgram-speech.provider.ts
            │   ├── funasr-speech.provider.ts
            │   └── faster-whisper.provider.ts
            ├── speech-provider.factory.ts
            ├── speech.service.ts
            └── speech.module.ts
```

如果后期语音能力变重，也可以从 `biz-service` 拆成独立 `speech-service`。

### 6.3 Provider 接口设计

```ts
export interface SpeechRecognitionProvider {
  readonly name: string;

  recognizeShortAudio(input: ShortAudioInput): Promise<SpeechRecognitionResult>;

  recognizeStream?(
    input: StreamAudioInput,
  ): AsyncIterable<SpeechRecognitionPartialResult>;

  transcribeFile?(input: FileTranscribeInput): Promise<SpeechRecognitionResult>;
}
```

类型建议：

```ts
export interface ShortAudioInput {
  audioUrl?: string;
  objectKey?: string;
  buffer?: Buffer;
  format: 'wav' | 'mp3' | 'm4a' | 'pcm' | 'opus';
  sampleRate?: number;
  language?: string;
  region?: 'CN' | 'GLOBAL';
  scene?: SpeechScene;
}

export interface FileTranscribeInput {
  audioUrl?: string;
  objectKey?: string;
  language?: string;
  scene?: SpeechScene;
}

export interface StreamAudioInput {
  sessionId: string;
  language?: string;
  scene?: SpeechScene;
}

export interface SpeechRecognitionResult {
  provider: string;
  text: string;
  language?: string;
  durationMs?: number;
  confidence?: number;
  segments?: SpeechSegment[];
  raw?: unknown;
}

export interface SpeechRecognitionPartialResult {
  text: string;
  isFinal: boolean;
  confidence?: number;
}

export interface SpeechSegment {
  startMs?: number;
  endMs?: number;
  text: string;
  confidence?: number;
}

export type SpeechScene =
  | 'meal_voice_record'
  | 'device_command'
  | 'voice_note'
  | 'health_assistant'
  | 'long_transcription';
```

## 7. Provider 路由策略

### 7.1 默认路由

```txt
CN_REGION:
  1. Tencent ASR
  2. Aliyun ASR
  3. iFlytek ASR
  4. Baidu ASR
  5. Local FunASR

GLOBAL_REGION:
  1. OpenAI gpt-4o-mini-transcribe
  2. Deepgram
  3. Azure Speech
  4. Google Speech-to-Text
  5. Local faster-whisper
```

---

### 7.2 按场景路由

```txt
meal_voice_record:
  CN      -> Tencent ASR
  GLOBAL  -> OpenAI mini-transcribe

device_command:
  CN      -> Tencent 一句话识别
  GLOBAL  -> OpenAI mini-transcribe

health_assistant:
  CN      -> Tencent ASR + LLM
  GLOBAL  -> OpenAI Realtime / Deepgram

long_transcription:
  CN      -> 讯飞 / 阿里云 / 百度
  GLOBAL  -> Deepgram / Azure / AssemblyAI

private_deployment:
  CN      -> FunASR
  GLOBAL  -> faster-whisper
```

---

### 7.3 失败兜底策略

建议每次请求带上：

- requestId
- userId / deviceId
- scene
- region
- primaryProvider
- fallbackProviders

示例：

```txt
CN meal_voice_record:
  primary: Tencent ASR
  fallback: Aliyun ASR
  final fallback: FunASR

GLOBAL meal_voice_record:
  primary: OpenAI mini-transcribe
  fallback: Deepgram
  final fallback: faster-whisper
```

注意：

- 设备短指令不建议多次自动重试太久，否则用户体验差。
- 长音频转写可以异步重试。
- 实时语音流失败后应快速断开并提示重连，而不是后台长时间重试。

## 8. 数据流设计

### 8.1 用户语音记录饮食

```txt
App / 小程序录音
  ↓
上传音频到 S3 / OSS
  ↓
调用 speech.recognizeShortAudio
  ↓
ASR 返回文本
  ↓
LLM 做饮食意图解析
  ↓
生成 meal item draft
  ↓
用户确认
  ↓
写入 meal record
```

---

### 8.2 设备语音指令

```txt
设备采集短语音
  ↓
上传音频 / 或边录边传
  ↓
ASR 识别短句
  ↓
命令意图解析
  ↓
映射为设备命令
  ↓
执行 / 下发 / 返回结果
```

示例指令：

```txt
开始称重
结束称重
重新识别
上传图片
查询今天热量
```

结构化命令：

```json
{
  "intent": "device_command",
  "command": "meal.weighing.start",
  "confidence": 0.96
}
```

---

### 8.3 实时健康助手

```txt
App 建立实时语音会话
  ↓
ASR 实时返回 partial text
  ↓
LLM 做上下文理解
  ↓
TTS 返回语音
  ↓
App 播放
```

该场景优先考虑：

```txt
Deepgram / OpenAI Realtime
```

## 9. MVP 实施计划

### P0：基础 Provider 抽象

目标：先不深度绑定厂商，完成统一接口和配置。

任务：

- 定义 `SpeechRecognitionProvider` 接口。
- 定义 `SpeechScene` 场景枚举。
- 定义统一 DTO 和返回结构。
- 增加 Provider Factory。
- 增加环境变量配置。

建议环境变量：

```env
SPEECH_DEFAULT_REGION=CN
SPEECH_CN_PROVIDER=tencent
SPEECH_GLOBAL_PROVIDER=openai
SPEECH_ENABLE_FALLBACK=true
```

---

### P1：国内 MVP 接入

目标：完成国内语音输入能力。

首选：

```txt
TencentSpeechProvider
```

任务：

- 接入腾讯云一句话识别。
- 接入腾讯云实时语音识别，视需求决定是否做。
- 支持音频 objectKey 从 OSS/S3 读取。
- 返回统一 `SpeechRecognitionResult`。
- 增加调用日志、耗时、错误码映射。

验收：

- 用户可通过语音输入饮食记录。
- 短语音识别耗时可接受。
- 识别失败能返回统一错误。

---

### P2：海外 Provider 接入

目标：支持海外或国际版使用。

首选：

```txt
OpenAISpeechProvider
```

任务：

- 接入 OpenAI `gpt-4o-mini-transcribe`。
- 支持多语言识别。
- 支持转写结果进入 LLM 结构化。
- 增加 OpenAI 调用成本统计。

验收：

- 英文、中文、混合语言均可识别。
- 饮食记录语音可以被结构化。
- 可通过配置切换 CN / GLOBAL Provider。

---

### P3：实时增强

目标：支持实时对话或实时字幕。

候选：

```txt
DeepgramSpeechProvider
OpenAIRealtimeSpeechProvider
```

任务：

- 支持 WebSocket 音频流输入。
- 支持 partial result。
- 支持 final result。
- 接入健康助手对话链路。
- 增加超时、断连、重连机制。

验收：

- 可以边说边出文字。
- 低延迟场景体验可接受。
- 会话断开后能安全释放资源。

---

### P4：私有化 / 降成本

目标：为大规模使用和私有化部署做准备。

候选：

```txt
FunASRProvider
FasterWhisperProvider
```

任务：

- 用 HTTP/gRPC 包装本地 ASR 服务。
- 支持 GPU/CPU 模式配置。
- 支持批量转写队列。
- 支持模型版本管理。
- 支持和云 Provider 的兜底互切。

验收：

- 私有化环境可不依赖外部 ASR 云服务。
- 大批量转写成本下降。
- 可监控 GPU/CPU 资源消耗。

## 10. 成本与性能策略

### 10.1 成本优化

建议：

1. 短语音优先使用一句话识别或低成本模型。
2. 长音频走异步转写，不占用同步请求。
3. 相同音频文件计算 hash，避免重复识别。
4. 对失败请求记录原因，不盲目多次重试。
5. 高并发后引入本地 ASR 服务降低边际成本。

---

### 10.2 性能优化

建议：

1. 设备指令类音频控制在 3～8 秒。
2. 饮食记录语音控制在 30 秒以内。
3. 长音频统一异步任务化。
4. 实时语音流必须支持 partial result。
5. 所有 Provider 调用都要设置超时时间。

建议超时：

| 场景 | 超时时间 |
|---|---:|
| 设备短指令 | 3～5 秒 |
| 用户短语音记录 | 5～10 秒 |
| 长音频转写 | 异步，不阻塞请求 |
| 实时语音流 | 建立连接 3 秒，空闲 30～60 秒断开 |

## 11. 风险点

### 11.1 国内外网络差异

OpenAI、Deepgram、Google、AWS 在国内网络链路上可能不稳定。

处理方式：

- 国内默认不用海外 Provider。
- 海外默认不用国内 Provider。
- Provider 根据部署区域动态选择。

---

### 11.2 方言和口音

饮食记录中可能出现方言、口语化、品牌名、菜名。

处理方式：

- 国内优先选择腾讯、讯飞、百度、阿里。
- ASR 后增加 LLM 纠错。
- 建立用户常用食物词库。
- 后期接入热词 / 自定义词表。

---

### 11.3 ASR 文本不能直接入库

ASR 结果只是自然语言文本，不应直接成为业务数据。

错误做法：

```txt
用户语音 -> ASR 文本 -> 直接写入 meal item
```

正确做法：

```txt
用户语音 -> ASR 文本 -> LLM 结构化 -> 用户确认 -> 写入 meal item
```

---

### 11.4 实时语音成本不可控

实时语音会话如果不限制时长，成本和资源都会不可控。

处理方式：

- 设置单次会话最大时长。
- 空闲自动断开。
- 对 C 端用户设置每日额度。
- 对 B 端或企业用户单独计费。

## 13. 最终架构一句话

语音识别模块在 Lumimax 中应作为 **Speech Provider Adapter 层**，负责把不同语音服务统一成标准文本结果；真正的业务价值不在 ASR 本身，而在后续的 **饮食语义理解、设备指令识别、健康意图结构化和营养分析联动**。
