# 第三方食品库接入调研

> 模块路径：`apps/biz-service/src/diet/`  
> 关联：[第三方食品营养识别调研.md](./第三方食品营养识别调研.md)（MVP 落地链路）、[food-query-v4.md](./food-query-v4.md)（当前实现基线）

---

如果让我从零设计一个**全球化食品识别 + 营养计算系统**，我不会一开始就把重点放在“接哪个食品库”，而是会设计成：

> **识别引擎负责判断“盘子里可能是什么”，食品数据库负责给“标准营养数据”，用户确认负责沉淀“真实吃法数据”。**

我的选型会分成 5 层。

---

# 1. 总体选型结论

我会选这个组合：

```text
识别层：
Passio / FatSecret Image Recognition / LogMeal 选 1-2 个测试

全球食品营养库：
FatSecret Platform API 作为全球主库

基础权威库：
USDA FoodData Central 作为基础营养基准

包装食品 / 条码：
Open Food Facts 作为全球免费条码库

文本 / 食谱 / 自然语言解析：
Edamam 作为营养解析与食谱能力补充

北美商业兜底：
Nutritionix 作为美国包装食品 / 餐厅食品补充

长期核心：
自建 Food Identity + 用户常吃库 + 家庭食品库 + 用户修正闭环
```

如果只选一个商业主库，我会优先看 **FatSecret**，因为它的定位更接近“全球食品营养平台”，官方说明覆盖 58 个国家、230 万以上 verified food items，并支持本地国家数据集。([Fatsecret 平台][1])

如果只选一个 AI 识别 SDK，我会优先测试 **Passio**，因为它把图片、语音、条码、文本输入和结构化营养结果放在一个 Nutrition-AI 平台里，官方文档也提到 2.5M+ food database。([passio.ai][2])

# 2. 我不会怎么做

我不会做成：

```text
图片识别 → 直接返回卡路里
```

也不会做成：

```text
一个第三方库 → 查不到就失败
```

这两个都不适合全球化。

全球化食品识别的难点是：

```text
同一个食物在不同国家叫法不同
同一个菜在不同地区做法不同
同一个名称可能对应完全不同的食物
包装食品依赖条码
家庭菜依赖用户修正
营养值依赖实际重量
AI 识别结果必须经过用户确认
```

所以架构必须是**多候选、多 Provider、多语言、多地区、自学习**。

# 3. 我会设计的识别流程

## 3.1 标准流程

```text
1. 用户 / 设备创建 meal
2. 设备上传 imageKey + weightGram
3. food-recognition 识别图片，返回 Top N 食物候选
4. food-normalizer 做语言、别名、地区标准化
5. nutrition-router 根据国家、语言、输入类型选择数据源
6. nutrition-providers 查询 FatSecret / USDA / Open Food Facts / Edamam / Nutritionix
7. nutrition-normalizer 统一成内部营养结构
8. ranking-service 对候选结果打分排序
9. 返回 Top 3-5 个候选给 App / 设备
10. 用户确认或修正
11. 保存 meal item
12. 沉淀到用户常吃库、家庭食品库、自建高频库
```

核心原则：

```text
AI 只给候选，不直接定案
数据库只给标准值，不代表用户真实吃法
用户确认才是最终事实
称重数据用于营养换算
修正数据用于长期提升准确率
```

# 4. Provider 选型

## 4.1 识别层：Passio / FatSecret Image Recognition / LogMeal

### 首选测试：Passio

Passio 适合做 App 端或云端食品识别能力测试。它官方描述支持 image recognition、barcode scanning、voice logging、food database 等能力，比较贴近“智能食品记录”这个场景。([passio.ai][2])

我会用它测试：

```text
图片识别
条码识别
语音食物记录
文本食物记录
营养结构化结果
移动端 SDK 体验
```

---

### 强烈建议测试：FatSecret Image Recognition

FatSecret 不只是营养库，它也提供图片识别能力。官方文档说明其 Image Recognition API 可以识别图片中的食物、估算 portion sizes 或 weights，并匹配到本地 verified country database。([Fatsecret 平台][3])

这点对全球化很关键，因为它不是“识别完再随便查一个全球库”，而是把识别结果映射到**本地国家食品数据集**。

---

### 备选测试：LogMeal

LogMeal 是专业食物图像识别方向，官方定位是通过 AI 判断食物及其营养特征。([LogMeal][4])

我会把它作为备选，而不是第一主选。原因是它适合测试图片识别能力，但是否能完整支撑全球化食品库、条码、文本、用户历史闭环，还需要商务和 API 能力进一步验证。

---

## 4.2 全球营养主库：FatSecret

如果我要做全球化，我会优先把 **FatSecret Platform API** 作为商业主库。

原因：

```text
覆盖国家多
有 verified food items
包含普通食品、品牌食品、包装食品、餐厅食品
支持本地国家数据集
也有图片识别和 NLP 能力
```

FatSecret 官方说明其 Platform API 覆盖 58+ 国家、230 万以上 verified food items，并提供本地 verified nutrition country datasets。([Fatsecret 平台][1])

我的定位：

```text
FatSecret = 全球商业营养主库
```

---

## 4.3 基础权威库：USDA FoodData Central

USDA 我一定会接，但不是作为全球主库，而是作为**基础食材权威基准库**。

USDA FoodData Central API 提供 REST 访问，目标就是帮助开发者将营养数据接入应用或网站；USDA 还提供 CSV 和 JSON 数据下载，适合做离线同步。([USDA 食品数据中心][5])

我的定位：

```text
USDA = 基础食材营养基准，不是全球商业食品主库
```

适合：

```text
apple
banana
egg
rice
chicken breast
beef
milk
flour
oil
vegetables
```

不适合作为唯一主库，因为它对全球本地菜、品牌食品、餐厅食品、亚洲家庭菜支持不够。

---

## 4.4 包装食品 / 条码：Open Food Facts

Open Food Facts 我也一定会接。

它适合做：

```text
条码扫描
包装食品
配料表
过敏原
Nutri-Score
NOVA 加工等级
欧洲和全球包装食品补充
```

Open Food Facts 官方教程展示了通过条码查询产品、获取 Nutri-Score 的场景；其项目介绍也说明它是食品产品数据库，包含配料、过敏原、营养事实和标签信息。([Open Food Facts][6])

我的定位：

```text
Open Food Facts = 全球免费包装食品 / 条码辅助库
```

但我不会把它作为唯一数据源，因为众包数据质量不稳定，需要做校验和评分。

---

## 4.5 文本 / 食谱 / 自然语言：Edamam

Edamam 我会接，但定位不是“全球主库”，而是：

```text
文本营养分析
食谱分析
自然语言食物描述解析
Recipe 能力
App 内容生态
```

Edamam 官方提供 Food Database API、Nutrition Analysis API、Recipe API；其 Nutrition Analysis API 可以返回热量、脂肪、碳水、蛋白质、胆固醇、钠等 28 类宏量和微量营养数据。([Edamam][7])

我的定位：

```text
Edamam = 文本 / 食谱 / 营养分析补充
```

它很适合处理：

```text
2 eggs with 100g rice
one bowl of chicken salad
homemade tomato pasta
recipe ingredient list
```

---

## 4.6 北美兜底：Nutritionix

Nutritionix 我会作为北美商业补充，而不是全球第一主库。

Nutritionix 官方强调其 verified nutrition database，并向 registered partners 提供数据库和 API。([Nutritionix][8])

我的定位：

```text
Nutritionix = 美国包装食品 / 餐厅食品 / UPC / 品牌食品补充
```

尤其适合美国市场，但如果目标是全球化，不能只依赖它。

# 8. Provider Router 怎么设计

我会做一个 `NutritionProviderRouter`，按输入类型和地区动态选择 Provider。

```ts
function resolveProviders(input: {
  countryCode?: string;
  locale?: string;
  hasBarcode?: boolean;
  hasImage?: boolean;
  hasText?: boolean;
  isRecipe?: boolean;
}) {
  if (input.hasBarcode) {
    return [
      'open_food_facts',
      'fatsecret',
      'nutritionix',
    ];
  }

  if (input.isRecipe || input.hasText) {
    return [
      'edamam',
      'fatsecret',
      'usda',
    ];
  }

  if (input.countryCode === 'US') {
    return [
      'user_common',
      'internal',
      'fatsecret',
      'nutritionix',
      'usda',
      'open_food_facts',
    ];
  }

  if (['FR', 'DE', 'ES', 'IT', 'NL', 'BE'].includes(input.countryCode ?? '')) {
    return [
      'user_common',
      'internal',
      'open_food_facts',
      'fatsecret',
      'edamam',
    ];
  }

  if (['JP', 'CN', 'KR', 'SG', 'MY', 'TH', 'VN'].includes(input.countryCode ?? '')) {
    return [
      'user_common',
      'internal',
      'fatsecret',
      'edamam',
      'open_food_facts',
      'llm_estimate',
    ];
  }

  return [
    'user_common',
    'internal',
    'fatsecret',
    'open_food_facts',
    'usda',
    'edamam',
    'llm_estimate',
  ];
}
```

# 9. 候选结果评分

每个 Provider 返回的结果都要统一打分。

```text
finalScore =
  userHistoryScore * 0.25
+ localeCountryScore * 0.20
+ nameMatchScore * 0.20
+ providerTrustScore * 0.15
+ imageConfidenceScore * 0.10
+ barcodeBrandScore * 0.05
+ nutritionCompletenessScore * 0.05
```

排序优先级：

```text
用户确认过 > 家庭食品库 > 自建高频库 > 本地国家 verified 数据 > 条码命中 > 名称相似 > LLM 估算
```

# 10. LLM 在这里怎么用

LLM 不应该作为主要营养数据库。

我会把 LLM 用在这些地方：

```text
1. 多语言食物名称标准化
2. 菜品拆解成可能食材
3. 第三方未命中时做粗略估算
4. 解释营养建议
5. 生成健康建议
6. 处理用户自然语言描述
```

例如：

```text
用户输入：一碗番茄鸡蛋面
LLM 拆解：
- noodles
- egg
- tomato
- oil
- broth
然后再分别查数据库和按重量估算
```

但 LLM 给出的营养值必须标记：

```text
sourceType = llm_estimated
verifiedLevel = unverified
confidence = low
```

不要把 LLM 估算结果和 verified food database 混在一起。

# 11. 最终架构策略

我会把系统做成三段式：

```text
第一段：识别
图片 / 语音 / 文本 / 条码 → 食物候选

第二段：匹配
候选食物 → 多 Provider 查询 → 标准营养数据

第三段：确认
用户确认 / 修正 → 保存 meal item → 沉淀自有数据
```

长期你的壁垒不是接了 FatSecret、Passio、Edamam，而是：

```text
用户真实吃过什么
用户怎么称重
用户怎么修正
同一家庭常吃什么
同一地区用户怎么叫这个食物
某个菜在当地通常怎么做
图片识别结果和用户确认之间的偏差
```
